새 판을 돌릴까?

부족분 ≥ 100장이면 find_batch_slot()로 생산일 산정. 미만이면 막고 사람 판단.

폭포는 위가 우선입니다 — 재고로 되면 생산 안 하고, 동승되면 새 판 안 벌립니다. "있는 것부터, 가장 싸게" 메우는 순서예요. 동승은 같은 품목 미충족 발주를 pp_order_schedule_status에서 가장 이른 납기로 한 건 찾고, 그 생산일(납기 − 리드타임 4일)에 추가 소요분을 얹어 17:00 정규시간에 들어가는지(day_used + 추가 ≤ 480분)까지 봅니다.

최소수량·부산물·소프트홀드·라인독립

신규배치 최소 100장

부족분 < 100이면 below_min — 짧은 런은 비경제적이라 엔진이 새 판을 안 만든다. 묶을지/그래도 돌릴지는 사람.

MIN_BATCH_QTY

앞뒤 부산물 +2

소요시간·용량은 shortfall + 2로 본다 — 기계 배치 앞뒤 자동 2장. 계획 target엔 안 더함(네팅).

3

소프트 홀드 (발주만)

발주 신규배치 후보일에 활성 일정문의가 있으면 다음 생산일로 최대 +1일만 양보. 문의가 발주를 무한정 못 막게.

SCHEDULE_DATE_SLACK

클러스터링

새 판을 돌릴 때 같은 계열·두께(±25mm)·심재 품목과 묶어서 한 날에. 월말이면 당월완결 우선.

라인은 독립 판정

멀티라인 문의는 judge_lines()가 라인마다 따로 judge(). 같은 카드 안 라인끼리 용량 누적은 안 함(제안일 뿐, 확정은 사람·라인별).

judge_lines
filter_center_focus
엔진은 두 층으로 나뉩니다 — 순수 계산(regular_minutes·item_minutes·find_batch_slot, DB 무관·grounded 대상)과 DB 오케스트레이션(judge(cur, …)). 라우터가 커서를 넘겨 호출하고, 용량 맵은 발주 스케줄을 백본으로 일별 소요분을 합산해 만듭니다.

5

카카오로 되돌려주기 — 2층 발송

즉시는 콜백, 지연은 알림톡

카카오엔 "아무 때나 자유 push"가 없습니다(무료·자유양식·API 중 2개만 택할 수 있는 삼각 트레이드오프). 그래서 타이밍으로 쪼갰어요.

시점메시지수단비용
즉시 (제출 직후)접수확인 · 부족분 되물음 · 일정문의 가능납기 자동응답콜백(챗봇 내)무료
지연 (직원 확정 후)발주완료 · 일정조율완료 · 일반답변 · 반려 · 일정문의 폴백알림톡 템플릿(API)≈ 월 천원
bolt
왜 즉시 콜백인데 CPU가 안 죽나? Cloud Run은 응답 직후 CPU를 거의 0으로 죽입니다. 그래서 webhook은 5초 안에 ACK만 던지고(useCallback) → Cloud Tasks가 백엔드를 실요청으로 다시 호출(=새 인바운드=풀 CPU) → 그때 Gemma를 12초까지 돌리고 카카오 콜백 URL로 결과를 푸시합니다. 덕분에 CPU 상시할당($0-idle 위반) 없이 무거운 LLM을 돌려요(plan 0057 ⑦ · 0058).
6

결정과 트레이드오프 — 왜 이렇게 지었나

카드를 눌러 채택 vs 기각
채택LLM은 수신 경계의 제안·정규화 소체로만. 출력=prefill 후보, 확정은 사람/결정규칙 뒤.
기각추출 결과로 바로 발주·판정 commit.
근거발주서가 이카운트(세무)로 흐른다. 오파싱 1건 = 오발주·오세무. 비결정성을 경계에 격리하면 본 레코드는 항상 검증된 값.
채택가용재고 → 미충족 발주 동승 → 신규배치 폭포(위 우선).
기각매 문의마다 무조건 새 배치 산정.
근거"있는 것부터, 가장 싸게". 재고 출고 > 기존 생산 합류 > 새 판이 비용 순. 잉여 생산·유령 슬롯을 안 만든다(demand-driven).
채택부족분 < 100이면 below_min — 새 배치 제안 안 함, 사람 판단.
기각잉여 생산으로 100장 채워 제안 / 50장짜리 배치 생성.
근거짧은 런은 비경제적. 엔진이 임의로 재고 적치를 만들지 않는다 — 묶을지/그냥 돌릴지는 demand 아는 사람이 결정.
채택일정문의는 슬롯 점유 X. 발주 배치가 활성 일정문의 날짜를 후순위로 최대 +1일만 양보.
기각일정문의가 생산슬롯을 실제로 점유.
근거점유하면 실주문 없는 유령 슬롯이 남는다. 신호는 남기되 캡(+1일)을 둬 문의가 발주를 무한정 못 막게. 넘는 충돌은 사람이 본다.
채택즉시=콜백(무료, 윈도우 ~1분), 지연=알림톡(API 정보성 템플릿).
기각콜백만으로 "처리완료"까지 / 상담톡(자유+API).
근거카카오 삼각 트레이드오프상 무료+자유양식+API 동시 불가. 콜백은 윈도우가 닫혀 지연 통지 불가 → 사람 확정 후 통지는 알림톡이 유일한 현실해. 상담톡은 유료 딜러계약이라 과설계.
채택반려도 완료류와 같은 알림톡 지연층(2026-06-16 정정).
기각제출 직후 즉시 콜백 자동반려(초기 가정).
근거반려 판단은 품목 확정·판정이 끝난 뒤 직원 인박스 지연 액션 → 콜백 윈도우 닫힘. 품목 자동 프리셀렉트로 신뢰가 오르면 즉시 콜백 자동반려로 승격(현재는 알림톡 폴백).

주의 + 다음에 볼 것

판정엔진을 만질 때
1

verdict는 제안

judge() 반환은 commit이 아니다. 확정(confirmed_kind/date)은 사람이 누른 그 1건만 DailyPlan에 기입.

2

폭포는 위 우선

새 분기 넣을 때 순서를 깨지 말 것 — 재고>동승>신규의 비용 순서가 엔진의 정체성.

3

유실 0 폴백

Gemma 장애·저신뢰면 None → 원문만 담긴 미파싱 카드. 고객 ACK는 나가고 직원이 수동·재파싱.

4

webhook은 토큰 게이트

챗봇 webhook은 CF Access가 아니라 X-Inquiry-Token 공유시크릿으로 검증(fail-closed). 토큰은 오픈빌더 스킬 설정에 수동 입력.

5

알림톡은 심사 의존

템플릿 미설정이면 configured()=False → no-op(로그만). 발송 코드는 그대로 돌아도 실제 발송은 심사 후.
