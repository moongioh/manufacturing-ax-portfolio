1차 — 다이어트 + 커밋 게이트

plan 0001 — 상태≠로그 · pre-commit 캡 · 역할 계약
1

상태 ≠ 로그

STATE.yaml은 "지금 스냅샷"만(166KB→스키마 준수 ≤16KB). 히스토리 서술은 WORKLOG.md로, WORKLOG는 회전. 같은 사실을 두 번 적지 않는다.

2

캡은 기계가 강제

git pre-commit 훅이 크기 초과를 커밋 단계에서 차단 — STATE >16KB·WORKLOG >64KB면 커밋 실패 + 안내. 에이전트 선의에 의존하지 않음. install-hooks.ps1로 양 repo 설치.

＋

로테이션 무손실

rotate-worklog.ps1: 오래된 날짜 섹션을 archive/worklogs/로 절단, 최근분만 잔류. "원본+아카이브 = 절단 전 전문" 보존 검증 내장.

3

죽은 규칙 0개

실존 artifact를 참조 안 하는 규칙은 삭제. 메타 규칙 신설: "규칙은 실존 파일/스크립트를 참조할 때만 유효 — 대상이 사라지면 규칙도 같이 지운다."

세션 역할 계약 — 병렬 세션이 같은 장부를 동시에 쓰면 레이스가 납니다. 쓰기 소유권을 역할별로 갈라 구조적으로 차단해요.

역할모델소유 (쓰기)세션 시작
S1 실무Opus프로젝트 코드 + 그 STATE 델타STATE + 활성 plan만(축소)
S2 기획·전역Fableplans·WORKLOG·메모리·거버넌스 MD·INDEX풀 프로토콜
S3 잡무Opus일회성 산출물만 (거버넌스 쓰기 금지)면제

2차 — 하네스로 내린 규칙

plan 0002 · 0004 — 세 개의 훅

0001이 git 쪽은 기계화했지만, 핵심 프로토콜은 여전히 "모델이 기억해서 지키는" 산문이었어요. Claude Code 하네스로 내릴 수 있는 규칙을 훅으로 전환합니다 — 단, 훅 수는 최소(훅 자체가 새 복잡도가 되면 본말전도).

1

SessionStart — 자동 주입

session-brief.ps1: 루트 WORKLOG 맵 + 활성 STATE.yaml + INDEX의 proposed/approved 행을 세션 시작에 자동 주입. "읽어라" 산문이 메커니즘으로 — 컴플라이언스 확률 → 1.

⚠

PreToolUse — 파괴 가드

guard-destructive.ps1: git reset --hard·DROP TABLE·TRUNCATE·alembic downgrade·Remove-Item -Recurse·gcloud … delete 등을 패턴 매칭으로 차단/ask 승급. prod DB 다루는 워크스페이스의 안전핀.

2

PostToolUse — 체크포인트 프록시

checkpoint-proxy.ps1(0004): STATE/WORKLOG를 쓰는 "작업 매듭"마다 트랜스크립트 크기 델타(~1.2MB)를 재서, 초과 시 /compact(연속)·/clear(전환) 넛지. append-only라 절대크기 대신 델타로(compact 후에도 정상).

3

메모리 대청소

40건 중 상태성 항목(~10건: "배포완료"·"미구현"·"다음=")을 STATE로 이관·삭제 → 인덱스 ~25줄. 썩은 메모리가 매 세션 잘못된 배경지식으로 주입되던 것을 끊음(State≠log를 메모리에도 적용).

4

단일 출처

같은 사실의 정본은 한 파일만, 나머지는 포인터. 공유 사실=GEMINI.md, Claude 전용=CLAUDE.md. 중복 발견 시 포인터화가 기본 동작.

report_problem
훅의 한계를 정직하게 못 박았어요(재논쟁 금지). 훅은 토큰 사용량을 못 읽고(입력 JSON에 토큰 필드 없음 → 크기 프록시가 유일), /compact·/clear를 실행도 못 합니다(텍스트 전달만 → 추천만, 실행은 사용자 1키). 그래서 SessionStart 훅이 리셋을 싸게 만드는 게 전제 — 상태가 자동 재주입되니 clear가 두렵지 않죠.

학습 모드 — 실행하면서 가르친다

plan 0005 → 0007

사용자는 코드/명령을 읽는 눈을 키우고 싶어 합니다. 에이전트가 그냥 빠르게 처리해 버리면 결과만 남고 학습이 0이에요. 그래서 이 워크스페이스의 기본 동작을 "실행하면서 가르치는" 모드로 바꿨습니다.

1

축① 작업 중 내레이션 (유지)

상태 변경·삭제·배포·DB·permission 유발 명령은 도구 호출 직전에 평어로 — 무엇을·왜 이 시점에·무엇을 주의. 단순 조회는 한 줄로 묶음(장황함 억제).

2

왜 훅이 아니라 행동 규약

자연어 교육은 명령마다 맥락이 달라 모델의 일이지 결정론적 훅이 정적 텍스트로 대체 못 함. 훅은 정해진 문구만 찍지 "왜 이 시점에 필요한지"를 못 가르친다.

↻

축② 종료 워크스루 → 0007로 대체

마감마다 강제하던 인라인(산문) 코드 워크스루는 폐지. 함수명·산문만으론 체화가 약했음(사용자 피드백) → 종종·기회적 인터랙티브 HTML 교육문서로 대체.

＋

지금 읽는 이 문서

주제(도메인) 단위·한글 파일명·깊이 3티어(A 풀 인터랙티브 / B 표준 도식 / C 요약). 이 거버넌스 워크스루가 바로 그 산물 — 강제 인라인이 약했던 자리를 채웁니다.

4차 — 검증단 두 층

plan 0081 · 0082 — 거짓과 미동작을 막는다

크기 게이트는 "용량"을 막지 "거짓"을 못 막아요. 실제로 한 번 청소해도 검출 장치가 없으니 STATE가 다시 썩었고 (점검 결과 next_steps 62%가 완료항목 = 스냅샷이 로그화), 에이전트는 썩은 STATE를 그대로 신뢰했습니다. 두 층의 검증단으로 짝을 채웁니다.

표면화지 차단 아님

3

2층 = 독립 적대 패널

검증됨 표식

결정과 트레이드오프 — 관통하는 원칙

채택 vs 기각 · 한눈에
쟁점채택기각근거
규칙 강제기계화 가능한 건 훅/캡으로전부 산문 규율"기억해서 지켜라"는 컴플라이언스 확률 <1. 기계는 1.
드리프트 대응표면화(세션 시작 경고)차단(커밋 막기)드리프트는 작업 중 정상 발생 → 차단은 오탐으로 일을 막음.
검출 vs 수정검출·표면화만, 수정은 사람완료항목 자동 퇴역✅ 휴리스틱 오탐 시 진짜 미완을 지움 — ERP 장부에서 치명적.
STATE 신선도 판정결정론 검사(같은 입력→같은 경고)LLM "현실과 맞나" 판정비싸고 비결정적이며 false confidence 생성. 의미 판단은 사람.
결정 기록처plans가 ADR 흡수docs/adr/ 신설plans의 번호·불변성·상태 모델이 이미 ADR 속성. 별도 디렉토리는 자동 로드 밖.
마무리 검증독립(fresh) 에이전트 — 마무리 한정자기 diff 검토만고비용 게이트라 확증편향 제거 가치 > 토큰비용. 평상 종료는 자기검토 유지.
학습 방식모델의 자연어 교육(행동 규약)명령별 자동 훅 문구"왜 이 시점에"는 맥락 의존 — 정적 텍스트가 대체 불가.
