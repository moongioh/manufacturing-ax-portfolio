온톨로지 확장 — Function · Action · GP

school교육 워크스루 · plan 0093 / 0110 / 0117 / 0119

개념그래프(Object·Link)는 "무엇이 무엇인가"를 압니다. 그 위에 두 층을 얹으면 그래프가 움직이기 시작해요 — Function(예측·계산을 붙이는 계약)과 Action(그래프를 바꾸는 타입드 쓰기). 시계열 예측(GP)과 팔란티어식 통합 콘솔이 이 확장의 이미 가동 중인 얼굴입니다. 핵심 규율은 변하지 않습니다 — 숫자는 코드가 진실, 온톨로지는 포인터.

핵심: 팔란티어식 온톨로지의 3요소 — Object(개념)·Function(예측/계산)·Action(쓰기)을 이 순서로 얹습니다. 기반 그래프와 예측(GP)·통합 콘솔은 이미 가동 중이고, Function/Action 레지스트리 계약은 이 둘을 1급 시민으로 정형화하는 설계 층입니다.

2

Function 레이어 — 공통 확률추정기 계약

plan 0093 · 계약 설계

"이 개념의 미래값은 어떻게 추정되나"를 개념마다 흩뿌리지 않고 하나의 계약으로 묶습니다.

1

공통 추정기 인터페이스

수요·현금흐름·소비 같은 추정 대상이 모두 같은 계약(입력 피처 → 분포 출력)을 따르게 — 엔진(고전 시계열 / 가우시안 프로세스)은 그 뒤에서 교체 가능.

2

object-type / function 레지스트리

어떤 개념 타입에 어떤 Function이 붙는지 레지스트리로 등록 — 개념그래프가 "나는 이렇게 예측된다"를 스스로 안내.

3

피처 lineage

추정에 쓰인 입력이 어느 데이터에서 왔는지 계보를 남겨, 예측의 근거를 역추적 가능하게.

psychology
엔진을 계약 뒤에 숨기는 이유: 지금 데이터에 어떤 추정기가 최적인지는 데이터가 쌓이며 바뀝니다. 계약을 고정하고 엔진을 갈아끼울 수 있게 하면, 모델 교체가 배관 공사가 아니라 부품 교환이 됩니다.
3

시계열 예측 레이어 — 이미 가동 중

plan 0117 · 구축

Function 계약의 첫 실물 — 현금흐름·수요·환경구동 원료소비를 시계열로 예측하는 레이어가 붙어 있습니다.

trending_up
데이터 게이트로 정직하게 운영합니다. 현재 데이터가 간헐적·럼피(드문드문)해서, 고전 통계 모델이 더 정교한 확률모델(가우시안 프로세스)을 실측에서 이겼습니다. 그래서 GP는 폐기가 아니라 데이터 게이트에 걸어둠 — 더 조밀한 데이터가 들어오면 계약을 통해 엔진만 교체해 재검토합니다. 모델 우월성을 주장하지 않고 측정으로 판단하는 게 이 레이어의 규율입니다.
history_edu
예측 피드는 라이브 DB가 아니라 장기 실측에서. 한 달치 라이브 데이터로 미래를 점치지 않고, 장기 운영 이력을 피드로 삼습니다 — 예측의 신뢰는 데이터의 길이에서 나온다는 원칙.
4

Action 레이어 — 타입드 거버넌스 쓰기

plan 0110 · 계약 설계

그래프를 바꾸는 모든 쓰기를 1급 Action으로 정형화합니다 — 무엇이 상태를 바꾸는지가 그래프에 드러나게.

Action 예바꾸는 것거버넌스
record_movement재고 이벤트(입출고·생산소비)단일 게이트 · 검증 통과분만
발주 승인발주 상태 전이권한(RBAC) · 감사 기록
입금 인식 · 분개채권·회계 상태사람 승인 게이트
shield_lock
타입드 = 아무 쓰기나 못 한다. Action을 타입으로 등록하면 각 쓰기가 어떤 개념을 · 어떤 권한으로 · 어떤 감사와 함께 바꾸는지가 계약으로 고정됩니다. 그래프가 "무엇이 나를 바꿀 수 있나"를 스스로 알게 되는 것 — 거버넌스가 데이터 모델의 일부가 됩니다.
5

팔란티어식 통합 콘솔 — 이미 가동 중

plan 0119 · 0103 · 구축

Object·Link·Function을 한 화면에서 탐색하는 관리콘솔 뷰 — 개념그래프 시각화(0103)를 확장해 통합했습니다.

1

Object / Link 뷰

개념 노드와 술어 엣지를 자동 레이아웃으로 시각화 — 도메인 구조를 눈으로 탐색.

2

Function 뷰

어떤 개념에 어떤 예측·계산이 붙어 있는지 통합 표시 — 그래프가 정적 사전이 아니라 살아있는 계기판이 됩니다.

3

디클러터 레이아웃

노드가 많아지면 엉키므로 자동 레이아웃 + 필터로 읽히게 — 시각화가 장식이 아니라 도구가 되도록.

6

결정과 트레이드오프 — 왜 이렇게 지었나

카드를 눌러 채택 vs 기각
채택자율화는 정합된 데이터 백본 위에만 얹는다. AI는 조언 표면 한정.
기각제로샷 LLM에 도메인 판단·숫자를 맡기기.
근거데이터가 흩어져 있으면 그 위의 자동화는 쓰레기를 빠르게 만든다. 온톨로지·검증으로 데이터를 먼저 정합화하고, 그 위에 예측·자동화를 얹어야 신뢰가 선다. AI는 도메인을 추측하지 않고 정본에서 착지한다.
채택모든 추정을 같은 계약 뒤에 두고 엔진 교체 가능.
기각수요·현금흐름·소비마다 별도 예측 코드를 직접 작성.
근거어떤 엔진이 최적인지는 데이터가 쌓이며 바뀐다. 계약을 고정하면 모델 교체가 부품 교환이 되고, 예측 대상이 늘어도 배관을 새로 깔 필요가 없다. 실제로 현 데이터에선 고전 모델이 이겨서 GP를 게이트에 걸어둘 수 있었던 것도 이 계약 덕분.
채택그래프는 포인터만 — 실제 값은 코드/레지스트리가 진실(SSoT).
기각추정 상수·계수를 개념 노드에 직접 저장.
근거같은 숫자를 두 곳에 두면 반드시 어긋난다(stale 드리프트). 값은 레지스트리에만 두고 그래프는 "여기를 보라"는 포인터만 갖는다. 그래야 값이 바뀌어도 그래프가 낡지 않는다 — 정합성은 단일 출처에서.
채택그래프를 바꾸는 쓰기를 1급 Action 레지스트리로 등록.
기각각 라우터가 자유롭게 DB를 직접 변경.
근거자유 쓰기는 어떤 코드가 무엇을 바꾸는지 추적 불가로 만든다. Action을 타입으로 등록하면 각 쓰기의 권한·감사·대상이 계약으로 고정돼, 거버넌스가 데이터 모델의 일부가 된다 — 자율화를 안전하게 이륙시키는 전제.
