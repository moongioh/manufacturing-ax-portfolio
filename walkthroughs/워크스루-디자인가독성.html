3단계 디자인 시스템 — 화면 성격을 나눈다

plan 0011

모든 화면을 한 가지 톤으로 만들면 업무 화면은 산만하고 문서 화면은 답답합니다. 성격에 따라 3개 계열로 분리했어요.

계열쓰임원칙
① ERP 업무발주보드·재고장·생산일지 등 일상 운영 화면정보 밀도 최대 · 인더스트리얼 블루 · 클릭 경로 최단
② 기술명세관리부 기술 부록·아키텍처 도식글 최소 · 도식 최대 · 전체폭
③ 창의 (교육)이 워크스루 문서·리포트여백·그라데이션·인터랙션 — 읽는 맛
palette
색은 하드코딩이 아니라 토큰입니다. 화면은 raw hex 대신 var(--primary)·var(--surface) 같은 의미 토큰만 씁니다. 그래서 다크모드 대응·팔레트 조정이 한 곳에서 끝나요(전 화면 일괄 재매핑). 대량 토큰 마이그레이션 없이도 테마가 흔들리지 않습니다.
2

가독성 스윕 — 재사용 가능한 레시피

plan 0122 → 0123 · 0095

발주보드에서 한 번 정립한 가독성 레시피를 핵심 화면들에 그대로 확장했습니다(0122 → 0123). 한 번 만든 규칙을 재사용.

1

상태 컬럼 그룹화

발주 상태를 흩뿌리지 않고 의미 그룹으로 묶어 스캔 가능하게 — "지금 뭘 봐야 하나"가 한눈에.

2

납기 위험 임계

납기가 임박·초과한 건을 색 임계로 자동 강조 — 위험이 먼저 눈에 들어옵니다.

3

편집모드 가시성 · 미정 칩

지금 편집 가능한지 UI가 명확히 신호(데이터 안전과 직결) · 미정 값은 회색 '미정' 칩으로 빈칸과 구분.

4

wide-fill 반응형

넓은 모니터에서 보고서가 가운데 좁게 몰리지 않고 화면을 유체적으로 채우도록 grid reflow(0095) — 넓은 화면의 여백 낭비 제거.

3

상태 피드백 하드닝 — 로딩·에러·빈 화면

상태처리이유
로딩스켈레톤 — 이전 데이터를 흐리게 남기지 않고 뼈대 표시(stale-while-loading 혼란 제거)"바뀐 건가 로딩인가" 모호함 제거
빈 데이터빈 상태 전용 안내 — 표가 그냥 비어 보이지 않게오해(장애?) 방지
에러정규화로 백엔드 null이 UI를 깨뜨리지 못하게 + 재시도 표면크래시 방지
알림(Toast)심각도별 색·아이콘(성공/경고/오류) 표준화중요도 즉시 전달
verified
모든 데이터 훅은 정규화를 거칩니다. 백엔드 응답이 상태로 들어가기 직전에 normalize*가 결측 필드·타입을 안전 기본값으로 강제 — 백엔드의 null이 화면을 크래시시키는 클래스를 원천 차단합니다. 로딩/빈/에러 처리는 그 위의 사용자 피드백 계층.
4

검색 자연화 — 사람이 치는 대로 찾는다

plan 0147 · 0148

거래처·품목 검색이 정확한 철자를 요구하면 현장은 느려집니다. 사람이 실제로 치는 방식에 맞췄어요.

초성 매칭 : "ㅅㅇ" → "삼원"   ·   접두 우선 : "대" → "대명…"이 먼저
한글 초성 분해 + 접두 우선 랭크. 자유 텍스트 클라이언트 즉시 필터라 서버 왕복 없이 반응.
join_inner
매칭 엔진을 공용화했습니다(0148). 각 검색(거래처·품목·지역)은 자기 UI를 유지하되, 초성+접두 로직만 공유합니다. 거대한 단일 검색 컴포넌트로 통일하는 과설계는 기각 — 데이터·렌더가 화면마다 다르기 때문. 로직만 나누고 UI는 각자.
5

프라이버시 마스킹 — 라이브 시연 안전

plan 0120
visibility_off
화면공유·외부 시연용 마스킹 모드. 거래처명·금액 등 민감 값을 라이브 화면에서 가려, 실제 운영 데이터를 띄운 채로도 안전하게 시연할 수 있습니다. 포트폴리오·데모의 영업비밀 노출을 표면 레벨에서 차단하는 스위치.
bookmark
데모는 한 걸음 더 갑니다. 공개 포트폴리오 데모는 마스킹에 의존하지 않고 더미 데이터 + 백엔드 없는 정적 빌드라 영업비밀 노출이 물리적으로 0 — 마스킹은 실데이터 라이브 시연용, 정적 데모는 완전 격리용으로 역할이 나뉩니다.
6

결정과 트레이드오프 — 왜 이렇게 지었나

카드를 눌러 채택 vs 기각
채택화면은 의미 토큰(--primary·--surface)만 사용.
기각화면마다 hex 색을 직접 지정.
근거hex를 흩뿌리면 다크모드·팔레트 변경 때 전 화면을 뒤져야 한다. 토큰은 한 곳에서 재매핑하면 전체가 일관되게 따라온다. 하드코딩된 색·다크 대비 실패 지점만 골라 교정하는 편이 대량 마이그레이션보다 안전(churn 최소).
채택업무·기술명세·창의 3계열로 성격 분리.
기각모든 화면을 하나의 통일 톤으로.
근거고밀도 업무 화면과 읽는 문서는 최적 밀도가 정반대다. 한 톤으로 강제하면 한쪽은 산만하고 한쪽은 답답해진다. 성격별 계열이 각 맥락의 가독성을 최대화한다 — 단, 색 토큰은 공유해 브랜드 일관성 유지.
채택로딩 중 스켈레톤 뼈대 표시.
기각이전 데이터를 흐리게 남긴 채 로딩(stale-while-loading).
근거이전 데이터를 흐리게 두면 사용자가 "값이 바뀐 건가, 로딩 중인가"를 구분 못 한다 → 오판. 스켈레톤은 "지금 불러오는 중"을 명확히 신호해 그 혼란을 제거한다.
채택초성+접두 매칭을 클라이언트에서 즉시 필터.
기각매 타이핑마다 서버에 검색 요청.
근거거래처·품목 목록은 한 번 로드하면 클라이언트에 다 있다 → 타이핑마다 왕복할 이유가 없다. 즉시 필터가 훨씬 반응적이고 서버 부하도 0. 목록이 아주 커지면 서버 초성 컬럼으로 회귀하는 임계만 정해두면 된다.
